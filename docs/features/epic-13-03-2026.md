# EPIC: Оптимизация загрузки и открытия дней задач

## Контекст

Пользователь замечает задержку при открытии любого дня, даже при небольшом объеме данных (менее 1000 задач). Текущий подход загружает и перерабатывает данные неравномерно: есть полные ревалидации, повторные вычисления и фоновые запросы, которые создают ощущение «тормозов» в момент навигации по дням.

## Цель эпика

Сделать открытие/переключение дня быстрым и предсказуемым за счет:

- начальной загрузки ограниченного окна дат (6 месяцев назад + 6 месяцев вперед),
- ленивой догрузки диапазонов при навигации календаря,
- точечной ревалидации после действий вместо полного перезапроса,
- контроля фоновых запросов, влияющих на responsiveness UI.

## Не-цели

- Полная переработка storage-слоя или миграция БД.
- Изменение UX-паттернов календаря/списка/канбана.
- Глобальный рефактор всех store одновременно.

## Продуктовый результат

- Открытие дня ощущается мгновенным (без заметной «паузы»).
- Скролл/переход по датам не блокирует интерфейс.
- После CRUD-операций данные актуальны без дорогой полной перезагрузки.

## Метрики успеха

- P95 времени открытия дня: < 100 мс (рендер + данные из кэша).
- P95 догрузки нового диапазона (за пределами окна): < 300 мс.
- Количество полных `getDays()` без диапазона: стремится к 0 в runtime.
- Количество IPC-запросов при смене дня: снижение минимум на 60%.

## Гипотезы узких мест

- Частые полные ревалидации списка дней после операций (`revalidate` грузит весь диапазон).
- Дорогие derived-вычисления при каждом переключении дня.
- Фоновые cross-branch проверки по активному дню конкурируют за ресурс main/storage и ухудшают latency UI.

## Архитектурное решение

### 1) Range-based cache в `tasks.store`

- Вводится `loadedRange = {from, to}` для текущей ветки.
- Инициальная загрузка: `[today-6m, today+6m]`.
- `setActiveDay(date)`:
  - моментально переключает activeDay,
  - асинхронно проверяет, близко ли дата к краю диапазона,
  - при необходимости догружает недостающий сегмент (слева/справа).
- Слияние данных по дню (upsert по `date`) без полного сброса массива.

### 2) Ленивое расширение окна

- Буфер у края (например, 14-21 день).
- Если активная дата попадает в буфер, догрузить дополнительный chunk (например, +3 месяца в соответствующую сторону).
- Избегать повторной догрузки одинаковых диапазонов (дедупликация in-flight запросов).

### 3) Точечная ревалидация

- После операций менять только затронутые дни:
  - create/update/toggle/delete: ревалидация 1 дня,
  - move between days: ревалидация 2 дней,
  - move across branch: удаление локального task + точечная ревалидация дня/ветки.
- Полную ревалидацию оставлять только для аварийных кейсов и sync-событий.

### 4) Оптимизация фоновых запросов в header

- Убрать агрессивные cross-branch проверки на каждую смену дня.
- Перевести на модель:
  - lazy при открытии branch-picker,
  - или кеш по ключу `branchId+date` с TTL,
  - или батч API-метод «активность по веткам на дату».

### 5) Наблюдаемость

- Добавить `performance.mark/measure` вокруг:
  - `setActiveDay` -> first meaningful paint контента,
  - fetch missing range,
  - revalidate paths.
- Добавить debug-счетчики: cache hit/miss, range extend count, full reload count.

## План работ (декомпозиция)

- Включить локальные замеры перформанса (dev-only).
- Снять baseline сценарии:
  - запуск приложения,
  - 20 переключений дней,
  - CRUD+move операции,
  - переключение веток.
- Зафиксировать метрики в документе сравнения.

Deliverables:

- Таблица baseline метрик (P50/P95).
- Список hot paths с вкладом во время.

## Phase 1. Range cache и ленивый prefetch

- В `tasks.store` реализовать состояние диапазона и API загрузки диапазона.
- Инициализировать окно 12 месяцев вокруг today.
- Добавить `ensureRangeForDate(activeDay)` при навигации.
- Реализовать merge/upsert по дням и дедупликацию запросов.

Deliverables:

- Переключение дня внутри окна не вызывает сетевой/IPC fetch.
- Выход за окно вызывает только догрузку missing-части.

## Phase 2. Точечная ревалидация операций

- Переписать `revalidate`-ветки CRUD/move на scoped refresh.
- Ввести хелперы:
  - `refreshDay(date, branchId?)`,
  - `refreshDays([date1, date2], branchId?)`.
- Сохранить fallback на full reload при ошибке консистентности.

Deliverables:

- Полный `getDays` не используется после каждой операции.
- Данные остаются корректными при reorder/move.

## Phase 3. Cross-branch checks (header)

- Перевести на lazy/cached режим.
- Добавить ограничение частоты и отмену устаревших запросов.
- Опционально: подготовить batched endpoint на main стороне.

Deliverables:

- Смена дня не триггерит N запросов по количеству веток.
- Badge по другим веткам остается функциональным.

## Phase 4. Stabilization

- Добавить unit tests для range merge и scoped revalidate.
- Добавить интеграционные smoke-сценарии на календарную навигацию.
- Обновить документацию по data-loading стратегии.

Deliverables:

- Тесты проходят стабильно.
- Метрики after >= target.

## Критерии приемки (Acceptance Criteria)

- При навигации по дням внутри загруженного окна UI не показывает загрузочный спиннер.
- При выходе за окно догрузка не блокирует взаимодействие с уже показанным днем.
- После create/update/delete/move данные на затронутых днях консистентны без полного reload.
- P95 открытия дня < 100 мс на тестовом профиле с 1000 задач.
- Нет регрессий в фильтрах, drag-and-drop, переключении веток и поиске.

## Риски и меры

- Риск: рассинхронизация кэша диапазонов.
  - Мера: fallback full reload + метрика рассинхронизации.
- Риск: race condition при быстрых переключениях дат.
  - Мера: request token / latest-wins и in-flight дедупликация.
- Риск: рост памяти из-за большого окна.
  - Мера: ограничить max-window, опционально LRU по диапазонам.
- Риск: сложность поддержки.
  - Мера: выделить явный модуль range loader и покрыть тестами.
