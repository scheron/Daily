# 05 — Risk Register: SQLite Migration

| ID   | Description                                                                                                                                                                                                                             | Likelihood | Impact | Mitigation                                                                                                                                                                                                                                           |
| ---- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R-01 | **Data loss during PouchDB→SQLite migration.** Incorrect field mapping, skipped records, or corrupted base64 decoding could lose user data.                                                                                             | Medium     | High   | Migration runs in single SQLite transaction — atomic rollback on failure. PouchDB data kept intact after migration. `.migrated` flag only written on success. Retry on next launch if failed.                                                        |
| R-02 | **`better-sqlite3` native module fails with electron-builder.** N-API binding may not survive ASAR packaging or code signing. Current `afterPack-mac.js` may need modification.                                                         | Medium     | High   | Validate in Phase 0 before any other work. Use `electron-rebuild` for correct binary. Test full build+sign+launch cycle on arm64 before proceeding.                                                                                                  |
| R-03 | **iCloud sync snapshot incompatibility between PouchDB and SQLite app versions.** User updates one Mac but not the other. PouchDB device reads v2 snapshot it can't parse.                                                              | Medium     | High   | v2 snapshot includes `version: 2` discriminator. PouchDB app on v1 should fail gracefully on unknown format (current `isValidSnapshot` returns false → no overwrite). Add explicit version field check in v1 reader to show "please update" message. |
| R-04 | **Corrupted snapshot during iCloud upload/download.** Partial read returns truncated JSON. Without `NSFileCoordinator`, no guarantee of read consistency.                                                                               | Medium     | Medium | Atomic write via temp+rename prevents corrupt writes. Read retry with exponential backoff (3x: 500ms, 1s, 2s). JSON.parse validation — if corrupt, return null and don't overwrite remote. Full `NSFileCoordinator` deferred to future epic.         |
| R-05 | **Orphan asset files accumulate on disk.** Files written to `assets/` but SQL INSERT fails, or files remain after task deletion without cleanup running.                                                                                | Low        | Medium | Orphan cleanup runs on app start: `readdir(assets/)` + `SELECT id FROM files` → delete unmatched. Soft-deleted file metadata cleaned after 7-day TTL.                                                                                                |
| R-06 | **Migration UX for large databases.** Users with 5000+ tasks and many file attachments may experience multi-second blocking during first launch after update.                                                                           | Medium     | Low    | Show splash/progress screen during migration (product decision needed). Log progress. Worst case: app appears frozen for a few seconds — acceptable for one-time operation.                                                                          |
| R-07 | **`sort_order` column introduces scope creep.** Epic schema adds `sort_order INTEGER` to tags table, but current `Tag` domain type has no such field. Implementing tag ordering could cascade to renderer.                              | Low        | Low    | `sort_order` column is added in schema (nullable, no NOT NULL). Not exposed in `Tag` domain type. Used internally for ORDER BY only. No renderer changes needed. Can be exposed in a future feature.                                                 |
| R-08 | **Race condition between manual and scheduled sync.** Even with AsyncMutex, queued syncs may execute back-to-back, doubling iCloud read/write operations.                                                                               | Low        | Low    | AsyncMutex queues rather than skips — second sync will find no changes (hashes match) and return early. Minimal overhead. `withElapsedDelay(1000)` prevents rapid visual status flicker.                                                             |
| R-09 | **`TaskInternal` type confusion.** With join tables, `TaskInternal` (tags as `ID[]`) is used at the service-model boundary. If eliminated too early, it creates a refactoring cascade.                                                  | Low        | Low    | Keep `TaskInternal` for now. Models accept `Partial<TaskInternal>` for updates (tags as ID arrays). Services convert `Tag[] → ID[]` before calling models. Evaluate removal in Phase 5 cleanup.                                                      |
| R-10 | **`daily://file/{id}` protocol performance regression.** Switching from buffer-in-memory (base64 decode) to `fs.readFile` from disk adds I/O latency.                                                                                   | Low        | Low    | Disk reads for local files are sub-millisecond on SSD/APFS. Eliminates base64 decode overhead. Net performance should be better. Stream support available if buffer approach proves insufficient.                                                    |
| R-11 | **Snapshot v2 hash incompatibility.** v1 hash strips `_attachments` and sorts by `_id`. v2 hash has no `_attachments` and sorts by `id`. Different hashes for logically equivalent data forces unnecessary push after v1→v2 conversion. | Medium     | Low    | Expected and acceptable: first sync after upgrade will push a v2 snapshot. This is a one-time cost. After both devices upgrade, hashes align normally.                                                                                               |
| R-12 | **PouchDB kept after migration consumes disk space.** LevelDB files for PouchDB remain until Phase 5 explicit cleanup.                                                                                                                  | Low        | Low    | PouchDB data is typically small (tens of MB). Cleanup in Phase 5 deletes PouchDB directory. Until then, disk space impact is negligible.                                                                                                             |
